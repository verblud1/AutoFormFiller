# Руководство по участию в проекте AutoFormFiller

## Обзор

Спасибо за интерес к участию в проекте AutoFormFiller! В этом документе описаны правила и рекомендации по участию в разработке проекта.

## Структура проекта

Ознакомьтесь с файлом [PROJECT_STRUCTURE.md](PROJECT_STRUCTURE.md) для понимания организации кода.

## Технологии

Проект использует следующие технологии:
- Python 3.7+
- CustomTkinter для GUI
- Selenium для автоматизации браузера
- Pandas для работы с Excel файлами
- OpenPyXL для чтения XLSX файлов

## Стиль кодирования

### Python
- Используем стиль кодирования PEP 8
- Максимальная длина строки: 120 символов
- Используем docstrings для документирования функций и классов
- Используем type hints для аннотации типов

### Именование
- Переменные и функции: `snake_case`
- Классы: `CamelCase`
- Константы: `UPPER_SNAKE_CASE`

### Docstrings
Используем Google-style docstrings:

```python
def example_function(param1: str, param2: int) -> bool:
    """Краткое описание функции.
    
    Args:
        param1: Описание первого параметра
        param2: Описание второго параметра
    
    Returns:
        bool: Описание возвращаемого значения
    """
    pass
```

## Процесс разработки

### 1. Создание Issue
Перед началом работы создайте issue с описанием задачи:
- Четко опишите проблему или функцию
- Укажите приоритет
- Назначьте себя (если возможно)

### 2. Создание ветки
Создайте ветку от `main` с понятным названием:
```
feature/имя-функции
bugfix/описание-бага
enhancement/описание-улучшения
```

### 3. Разработка
- Следуйте стилю кодирования проекта
- Пишите чистый, понятный код
- Добавляйте комментарии к сложным алгоритмам
- Пишите unit-тесты для новых функций

### 4. Тестирование
- Протестируйте вашу функцию вручную
- Убедитесь, что она работает корректно
- Проверьте, что не сломали существующий функционал

### 5. Pull Request
- Опишите изменения в PR
- Ссылайтесь на связанное issue
- Убедитесь, что код проходит все проверки
- Запросите ревью у других разработчиков

## Типы изменений

При создании commit сообщений используйте следующие префиксы:

- `feat:` - новая функция
- `fix:` - исправление бага
- `docs:` - изменения в документации
- `style:` - изменения стиля (без изменения логики)
- `refactor:` - рефакторинг кода
- `test:` - добавление или изменение тестов
- `chore:` - прочие изменения

## Отладка

### Запуск в режиме отладки
Для отладки используйте режимы запуска:
```bash
# Запуск создателя JSON файлов
python main.py --mode creator

# Запуск массового обработчика
python main.py --mode processor
```

### Логирование
Используйте встроенные механизмы логирования:
```python
from utils.logger import log_message

log_message("Сообщение для лога")
```

## Тестирование

### Модульные тесты
При добавлении новых функций создавайте соответствующие unit-тесты:
```python
def test_example_function():
    assert example_function("test", 1) == True
```

### Интеграционные тесты
Проверяйте взаимодействие между модулями.

## Документация

### Код
- Добавляйте docstrings ко всем публичным функциям и классам
- Комментируйте сложные участки кода
- Обновляйте существующую документацию при изменениях

### Проект
- Обновляйте README.md при добавлении новых функций
- Обновляйте EXAMPLES.md с примерами использования
- Обновляйте API.md при изменении интерфейсов

## Безопасность

- Не храните конфиденциальные данные в коде
- Используйте безопасные методы работы с файлами
- Проверяйте вводимые пользователем данные

## Поддержка

### Сообщения об ошибках
При обнаружении бага:
1. Проверьте, не создано ли уже issue
2. Создайте новое issue с подробным описанием
3. Укажите шаги для воспроизведения
4. Приложите скриншоты или логи, если возможно

### Запросы функций
При предложении новой функции:
1. Объясните, почему она необходима
2. Опишите предполагаемую реализацию
3. Укажите возможные проблемы и решения
